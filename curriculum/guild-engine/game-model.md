# Guild Engine Game Model

## Status

This file defines the reusable gamification engine. Course folders adapt this model to their own content.

## Core loop

**Train -> Attempt -> Feedback -> Revise -> Log Evidence -> Level Up**

Every course version should preserve this loop.

The loop means:

1. Practise a target skill.
2. Try a task that uses the skill.
3. Get feedback.
4. Improve, reread, revise, or retry.
5. Log evidence of what changed.
6. Earn XP, rank progress, or a checkpoint.

## Student role

Students are guild members. They complete quests, practise skills, collect evidence, and rank up through demonstrated growth.

Students should know:

- what skill they are training
- what evidence will show progress
- what next move will help them improve

## Teacher role

The teacher is the game master and instructional coach.

The teacher controls:

- quest goals
- evidence standards
- pacing
- feedback
- rank-up approval
- conferences
- privacy boundaries
- when to simplify mechanics

The system should reduce management load. If point tracking takes over the class, the model is too heavy.

## Required game objects

- Quest: structured learning challenge.
- Training task: practice before evidence is submitted.
- Evidence: proof of skill, strategy use, revision, or growth.
- XP: credit for productive learning behaviours.
- Rank: progress inside a learning path.
- Level: broad campaign progress.
- Boss task: larger demonstration after training.
- Reflection: student explanation of what changed and what comes next.

## Optional game objects

Use only when helpful:

- character classes
- parties
- badges
- inventory or tool cards
- class milestones

These should stay light. The game layer supports instruction; it does not replace it.

## What the model rewards

- focused practice
- strategy use
- feedback use
- rereading or retrying
- revision
- improvement from a personal baseline
- accurate reflection
- useful collaboration
- evidence of skill

## What the model does not reward

- ranking students against each other
- raw result alone
- speed alone
- compliance without learning evidence
- task completion that hides misunderstanding

## XP rules

XP should be small, predictable, and tied to the loop.

Recommended XP categories:

- Training XP
- Strategy XP
- Feedback XP
- Evidence XP
- Reflection XP
- Growth XP
- Support XP

XP should not be mathematically complex.

## Rank-up rules

A rank-up requires:

1. a named skill
2. a clear evidence standard
3. student evidence
4. teacher confirmation
5. a next goal

Ranks are stronger than XP because they require evidence.

## Quest structure

A reusable quest should include:

1. title
2. learning target
3. short narrative hook
4. training task
5. evidence task
6. optional party task
7. boss or mini-boss task
8. reflection prompt
9. possible XP
10. possible rank evidence

Narrative should be light. The task must remain clear.

## Daily operating model

Recommended daily flow:

1. Daily Encounter
2. Guild Lesson
3. Training Rotations
4. Solo Evidence
5. Character Log

## Weekly operating model

Recommended weekly flow:

1. Launch the quest.
2. Train the target skill.
3. Practise with support.
4. Build evidence.
5. Complete a boss task or level-up reflection.

## Adaptation rules

Keep:

- core loop
- quests
- evidence
- XP
- ranks
- boss tasks
- reflection
- privacy rules

Replace:

- learning paths
- evidence types
- course language
- quest content
- skill standards

Example path sets:

- Literacy: Voice, Insight, Wordcraft
- Science: Observe, Explain, Apply
- Social Studies: Evidence, Perspective, Cause and Consequence
- Writing: Ideas, Structure, Style
- Design: Plan, Build, Improve

## Minimum viable version

The first classroom-ready version needs:

- three learning paths
- one XP menu
- one rank-up ladder per path
- one weekly quest template
- one boss task template
- one evidence log
- one teacher tracker
- one student reflection routine
